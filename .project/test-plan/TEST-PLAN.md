## 1. 测试策略

### 1.1 测试类型

| 类型 | 说明 | 工具/方法 |
|:-----|:-----|:-----------|
| 单元测试 | Services 层核心函数 | Vitest + Mock |
| API 测试 | REST API 端点验证 | curl / Postman / Jest |
| UI 测试 | 页面组件渲染和交互 | 手动测试 + 截图对比 |
| E2E 测试 | 关键用户流程 | Playwright / Cypress |

### 1.2 测试层级

```
┌─────────────────────────────────────┐
│         E2E 测试 (手动)             │  ← 关键用户流程
├─────────────────────────────────────┤
│         API 测试 (自动化)            │  ← 数据正确性
├─────────────────────────────────────┤
│         UI 测试 (手动)              │  ← 界面和交互
├─────────────────────────────────────┤
│         单元测试 (Vitest)            │  ← 核心逻辑
└─────────────────────────────────────┘
```

### 1.3 测试数据来源

| 数据 | 来源 | 说明 |
|:-----|:-----|:-----|
| 项目数据 | `$PROJECTS_DIR/*/.project/status.json` | 扫描所有项目 |
| 任务数据 | `.project/tasks/*.json` | 任务文件 |
| 测试结果 | `.project/test-results/` | Markdown 报告 |
| Bug 数据 | `.project/test-results/bugs/` | Bug 报告 |

## 3. Bug 追踪测试方案

### 3.1 数据来源

Bug 数据从 `.project/test-results/bugs/` 目录读取：

```
test-results/
├── bugs/
│   ├── BUG-001.md    # Markdown 格式 Bug 报告
│   └── index.json    # Bug 索引（汇总）
```

### 3.2 Bug 数据结构

```json
{
  "id": "BUG-001",
  "title": "Bug 标题",
  "severity": "critical|major|minor|trivial",
  "status": "open|in_progress|fixed|closed",
  "created_at": "2026-03-20T10:00:00+08:00",
  "fixed_at": "2026-03-20T13:00:00+08:00",
  "assignee": "Engineer"
}
```

### 3.3 统计计算逻辑

| 指标 | 计算公式 |
|:-----|:---------|
| 总数 | count(all bugs) |
| 已修复 | count(status = fixed or closed) |
| 未修复 | count(status = open or in_progress) |
| 修复率 | (已修复 / 总数) * 100% |

### 3.4 测试验证点

1. **Bug 列表读取** - 验证能读取 test-results/bugs/ 下的所有 Bug
2. **严重程度显示** - critical 红色、major 橙色、minor 黄色、trivial 绿色
3. **状态显示** - open 红、in_progress 黄、fixed 绿
4. **修复率计算** - 手动验证计算结果
